Claude Crew FAQ

Honest answers to tough questions

对尖锐问题的诚实回答

Competitive Comparisons

"Claude Code Remote is Anthropic's official solution. Why would I use yours?"

Claude Code Remote (CCR) solves a different problem: it lets claude.ai connect to a remote machine for execution. You still interact via the claude.ai web UI, one project at a time, one person at a time.

Claude Crew solves multi-project orchestration: 5 projects in parallel, team-shared timeline, phone-first control, task queue, cron jobs, permission system. CCR is a transport layer; Claude Crew is a management layer.

They're complementary. CCR connects a browser to a server. Claude Crew connects a Telegram group to N projects on that same server.
"Anthropic's Telegram Plugin already lets me chat with Claude via Telegram. How is this different?"

The official Telegram Plugin forwards messages between one Telegram chat and one Claude Code session. It's 1:1 — one chat, one project, one user.

Claude Crew is N:N — one group chat manages unlimited projects, each with its own isolated bot, independent context, separate permissions. You also get: task queue, cron scheduler, pinned dashboard, team access control, auto-restart daemon, context warnings, and an interactive button menu for all management.

Plugin = remote chat for one session. Claude Crew = cluster management for N projects.
"Happy Coder does Telegram control too, and it has a better UI."

Happy Coder is a great project with a polished chat UI. Key differences:

ContextGrows until crashAuto-warn 80%, auto-compact 95%, sessionMode
Happy Coder is good for single-project remote control. Claude Crew is built for multi-project cluster management.
"cc-connect also does multi-bot group management. What's the real difference?"

cc-connect takes a similar approach (multiple bots in one group). Differences:

  • Claude Crew: Full interactive button menus for all config, no terminal needed after setup. cc-connect requires terminal for management.
  • Claude Crew: Built-in resilience (circuit breaker, adaptive rate limiting, truncation recovery). cc-connect has basic error handling.
  • Claude Crew: Platform abstraction layer with Discord/Feishu planned. cc-connect is Telegram-only.
  • Claude Crew: sessionMode (continue/fresh), auto-compact, context warnings. cc-connect has basic session management.
Similar architecture, different polish level. Try both and see which fits your workflow.
"Why not just use tmux + Claude Code CLI? Zero dependencies, works perfectly."

If you work on one project, from one machine, alone — tmux works great. Claude Crew adds value when:

  • You manage 2+ projects and want to dispatch from one place
  • You want phone control — tmux on a phone is painful
  • You want push notifications when tasks finish
  • You have team members who need shared visibility
  • You want unattended operation — daemon runs 24/7, auto-restarts, cron jobs

If none of these apply, tmux is simpler. No shame in that.

Architecture Challenges

"Why Telegram? It's blocked in China and not popular in the US."

Telegram was the starting point because of its superior Bot API: reliable webhooks, inline buttons, no message rate limits, group privacy controls, free worldwide.

The architecture uses a Platform abstraction layer. Discord and Feishu adapters are planned. Adding a new platform = implementing one TypeScript interface. Core logic (permissions, queue, cron, dashboard) is platform-agnostic.

For China: Feishu (Lark) support is on the roadmap. For the US: Discord adapter is implemented and in testing.

"Spawning a new claude -p process for every task is wasteful. Why not keep a long-running session?"

Deliberate design choice. Short processes give you:

  • Context isolation — no bleed between tasks
  • Crash isolation — one bad task doesn't kill the daemon
  • Predictable cost — each task starts from a known context size
  • No context bloat — with sessionMode: "fresh", every task is clean

The trade-off: Claude re-reads project files each time (~10-30s overhead). For most tasks, this is acceptable. For rapid-fire micro-edits, use sessionMode: "continue" to accumulate context.

Short processes = predictable + resilient. Long sessions = faster but fragile. We default to predictable.
"No Docker isolation? That's a security joke for multi-user."

Fair criticism. Claude Crew runs all bots in a single process with filesystem access. The permission system (readOnly / approve / allowedUsers) provides control, not security boundaries.

This is by design for the target audience: solo developers and trusted small teams on personal machines. The daemon is meant to run on YOUR machine, managing YOUR projects.

For untrusted multi-tenant use: yes, you need Docker. Claude Crew is not designed for that scenario and says so explicitly in the README.

"approve mode runs Claude twice. That's literally doubling my cost."

Yes. The first run is read-only (Claude analyzes but can't write). If writes are needed, the second run retries with approved tools.

Why it's worth it: You see what Claude wants to do before it does it. In a team setting with production code, this prevents accidental damage.

Alternatives:

  • auto — Claude's safety classifier decides. No double run. Blocks genuinely dangerous ops.
  • allowAll — Everything pre-approved. No double run. Fastest, cheapest. Use for trusted personal projects.

Switch via button menu anytime. No restart needed.

"Context hits 90% after 3 tasks. How is that usable?"

With sessionMode: "continue" (default), each task accumulates on the previous one. Reading 3 files + editing 2 + running bash = ~30-50K tokens per task. Three tasks = 90%+ context.

Solutions (pick one):

  1. /new — reset context. Claude re-reads files from disk. ~10s overhead.
  2. /compact — compress history, keep key decisions. Lower overhead.
  3. sessionMode: "fresh" — every task starts clean. No accumulation. Relies on periodic memory files for continuity.

Auto-compact at 95% is built in. At 80%, the bot warns you proactively.

"Why does this project have 7000+ lines of TypeScript? Feels over-engineered."

Breakdown:

  • ~1200 lines: i18n (bilingual UI text for every button, message, and error)
  • ~1000 lines: interactive setup (button menus for config, bots, users, onboarding)
  • ~800 lines: daemon + bot-setup (Telegram handler registration, routing)
  • ~700 lines: claude.ts (invocation, progress, approval, resilience, context management)
  • ~600 lines: platform adapters (Telegram + Discord)
  • ~500 lines: config, types, helpers, state, cron, memory, dashboard
  • ~250 lines: resilience module

No framework, no abstraction layers for the sake of it. Every file does one thing. The largest file is i18n (bilingual strings). Remove i18n and it's ~5500 lines for a full management system.

Security Challenges

"Bot tokens stored in a plain JSON file? Really?"

bot-pool.json is stored with chmod 600 (owner read/write only). It's in ~/.claude/channels/telegram/, excluded from git via .gitignore.

This is the same security model as ~/.ssh/id_rsa, ~/.aws/credentials, and ~/.claude/ itself. If an attacker has read access to your home directory, you have bigger problems than bot tokens.

"Can other people in the group see my code?"

They see Claude's responses in the group chat — which may include code snippets, file contents, and diff outputs. They do NOT have direct filesystem access.

Control visibility:

  • Set allowedUsers per bot — unauthorized users can't even trigger the bot
  • Use readOnly for bots on sensitive projects — no write output to leak
  • Use a private group with only trusted members
  • For maximum isolation: one group per project (1:1 Focus Mode)
"ANTHROPIC_API_KEY is passed to Claude subprocesses. Isn't that dangerous?"

Claude Code CLI requires the API key to function. Without it, no task can run. The key is passed via filtered environment variables — everything except explicitly safe variables is blocked.

The risk: a prompt injection in your project files could instruct Claude to exfiltrate the key via a Bash command. Mitigations:

  • approve mode — you see and approve every Bash command
  • auto mode — Claude's safety classifier blocks exfiltration patterns
  • The key is already available to anyone with access to your ~/.claude/ directory

If this concerns you: use a scoped API key with spending limits, or use subscription auth instead.

Practical Questions

Is this really faster than just opening a terminal?

For a single task on a single project? No. Opening a terminal and running claude is faster.

Claude Crew is faster when:

  • You're managing 3+ projects and don't want to context-switch between terminals
  • You're away from your computer and want to dispatch from your phone
  • You want to check on all projects at a glance (dashboard)
  • You need unattended operation (cron jobs, auto-restart)
  • Your team needs shared visibility
How much does it cost to run?

Claude Crew itself is free and open-source. You pay for Claude API usage (tokens × model price).

Cost factors:

  • approve mode = 2× invocation cost (read-only + retry). Use allowAll or auto to halve it.
  • Screenshots = 50K+ tokens each. Prefer text descriptions.
  • Memory files grow over time and are loaded every session. Clean periodically.
  • Cron tasks are full invocations. Use longer intervals for non-urgent checks.

Cheapest config: model: "haiku" + sessionMode: "fresh" + permissionMode: "allowAll".

What platforms are supported?
PlatformStatusNotes
TelegramSupportedFull feature parity, production-tested
DiscordPlannedAdapter implemented, in testing
Feishu (Lark)PlannedArchitecture ready, adapter not started

Config is platform-segmented — switching platforms is a one-field change, no data loss.

What happens when my API key expires?

The circuit breaker detects consecutive failures, pauses the bot after 3 attempts, and tells you the root cause. Auto-recovers after 5 minutes. Update your key and the bot resumes — no restart needed.

Will this project be maintained?

Open-source, MIT license. The codebase is ~7200 lines of TypeScript with minimal dependencies (grammY + discord.js). It's small enough for one person to maintain or for anyone to fork and modify.

No cloud service dependency. No subscription. If the repo goes stale, your existing setup keeps running. The only external dependencies are Telegram's API and Anthropic's API — both stable, public APIs.

"Claude Code Remote 是 Anthropic 官方方案，为什么要用你的？"

Claude Code Remote (CCR) 解决的是另一个问题：让 claude.ai 网页连接到远程机器执行。你仍然通过浏览器操作，一次一个项目，一个人用。

两者互补。CCR 把浏览器连到服务器，Claude Crew 把 Telegram 群连到 N 个项目。
"Anthropic 官方的 Telegram Plugin 已经能用 Telegram 聊天了，你这个有什么不同？"

官方 Telegram Plugin 是 1:1 — 一个聊天窗口转发到一个 Claude Code 会话。一个项目，一个人。

Claude Crew 是 N:N — 一个群管理无限项目，每个项目独立 bot、独立上下文、独立权限。你还得到：任务队列、定时任务、看板、团队权限、自动重启、上下文预警、全套按钮菜单。

Plugin = 一个会话的远程聊天。Claude Crew = N 个项目的集群管理。

cc-connect 采用了类似的架构（多个 bot 在一个群里）。差异：

  • Claude Crew：完整的按钮菜单管理所有配置，setup 后不需要终端。cc-connect 需要终端管理。
  • Claude Crew：内置容错（熔断器、自适应限速、截断续写）。cc-connect 基础错误处理。

如果你只做一个项目、只在一台机器上、一个人用 — tmux 完全够用。Claude Crew 在以下场景有价值：

  • 想用手机控制 — 手机上用 tmux 很痛苦
  • 想要推送通知，任务完成手机响

如果这些都不需要，tmux 更简单。

架构质疑

"为什么选 Telegram？中国被墙，美国不流行。"

Telegram 的 Bot API 最适合这个场景：可靠的推送通知、内联按钮、消息无速率限制、群隐私控制、全球免费。

架构使用平台抽象层。Discord 和飞书适配器已在计划中。新增平台 = 实现一个 TypeScript 接口。核心逻辑（权限、队列、定时、看板）与平台无关。

中国用户：飞书支持在路线图上。美国用户：Discord 适配器已实现，测试中。

"每次任务都起新进程 claude -p，不浪费吗？"

刻意的设计选择。短进程带来：

  • 上下文隔离 — 任务之间不串
  • 崩溃隔离 — 一个任务崩溃不影响 daemon
  • 成本可预测 — 每次从已知大小的上下文开始
  • 无上下文膨胀 — sessionMode: "fresh" 每次都是干净的

代价：Claude 每次重新读项目文件（~10-30s 开销）。对大多数任务可接受。需要快速连续微编辑时，用 sessionMode: "continue" 累积上下文。

短进程 = 可预测 + 可靠。长会话 = 更快但脆弱。我们默认选可预测。
"没有 Docker 隔离？多人用的安全性是个笑话吧？"

批评有道理。所有 bot 跑在一个进程里，共享文件系统。权限系统提供控制，不是安全边界。

这是面向目标用户的设计：个人开发者和信任的小团队，跑在自己的机器上。daemon 就是在你的机器上管理你的项目。

需要不信任的多租户隔离？对，你需要 Docker。Claude Crew 不是为那个场景设计的，README 里明确说了。

"approve 模式跑两次 Claude，费用直接翻倍。"

是的。第一次只读（分析但不能写），第二次带权限重试。

为什么值得：你能在 Claude 动手之前看到它要做什么。团队场景下管理生产代码，这防止意外损坏。

替代方案：

  • auto — 安全分类器自动决定。不跑两次。阻止真正危险的操作。
  • allowAll — 全部预授权。不跑两次。最快最便宜。个人信任项目首选。

菜单里随时切换，不需要重启。

"3 个任务上下文就 90% 了，这怎么用？"

默认 sessionMode: "continue" 会累积上下文。读 3 个文件 + 改 2 个 + 跑 bash ≈ 30-50K tokens/任务。三个任务 = 90%+。

解法（选一个）：

  1. /new — 重置上下文。Claude 从磁盘重新读文件。~10s 开销。
  2. /compact — 压缩历史，保留关键决策。
  3. sessionMode: "fresh" — 每次独立上下文。无累积。靠 memory 文件恢复记忆。

95% 自动压缩已内置。80% 会主动提醒。

"7000 多行 TypeScript？感觉过度工程了。"

拆开看：

  • ~1200 行：i18n（中英双语 UI 文案）
  • ~1000 行：交互式设置（按钮菜单、配置、用户管理、引导）
  • ~800 行：daemon + 消息处理（Telegram handler、路由）
  • ~700 行：Claude 调用（执行、进度、审批、容错、上下文管理）
  • ~600 行：平台适配器（Telegram + Discord）
  • ~500 行：配置、类型、工具函数、状态、定时、记忆、看板
  • ~250 行：容错模块

没有框架，没有为抽象而抽象。每个文件做一件事。最大的文件是 i18n（双语字符串）。去掉 i18n 就是 ~5500 行的完整管理系统。

安全质疑

"Bot token 存在明文 JSON 里？这安全吗？"

bot-pool.json 存储权限为 chmod 600（只有 owner 可读写），在 ~/.claude/channels/telegram/ 目录下，通过 .gitignore 排除在 git 外。

和 ~/.ssh/id_rsa、~/.aws/credentials、~/.claude/ 本身的安全模型一样。如果攻击者能读你的 home 目录，bot token 不是你最大的问题。

"群里其他人能看到我的代码吗？"

他们能看到 Claude 的回复（可能包含代码片段、文件内容、diff 输出），但没有直接文件系统访问。

控制可见性：

  • 按 bot 设 allowedUsers — 未授权用户连触发 bot 都做不到
  • 敏感项目设 readOnly — 无写操作输出可泄露
  • 私人群只加信任的人
  • 最大隔离：一个群一个项目（1:1 Focus Mode）
"ANTHROPIC_API_KEY 传给了 Claude 子进程，不危险吗？"

Claude Code CLI 需要 API key 才能运行。没有它任何任务都跑不了。key 通过过滤后的环境变量传递 — 除了明确安全的变量，其他全部屏蔽。

风险：项目文件中的 prompt injection 可能指示 Claude 通过 Bash 命令外泄 key。缓解措施：

  • approve 模式 — 你审批每一个 Bash 命令
  • auto 模式 — 安全分类器阻止外泄模式
  • key 本身已经在你的 ~/.claude/ 目录里，有本地访问权的人都能拿到

如果仍然担心：使用带消费限制的 scoped API key，或用订阅认证代替。

实用问题

真的比直接开终端快吗？

单个项目单个任务？不一定。开终端跑 claude 更直接。

Claude Crew 更快的场景：

  • 管理 3+ 项目，不想在终端间切换
  • 不在电脑前，想用手机调度
  • 想一眼看到所有项目状态（看板）
  • 想要无人值守（定时任务、自动重启）
  • 团队需要共享可见性
运行成本多少？

Claude Crew 免费开源。你只付 Claude API 用量（tokens × 模型价格）。

最省配置：model: "haiku" + sessionMode: "fresh" + permissionMode: "allowAll"。

省钱技巧：简单任务用 haiku，避免发截图（单张 50K+ tokens），定期清理 memory 文件，非紧急定时任务用长间隔。

配置按平台分段 — 切换平台只改一个字段，数据不丢。

API key 过期了怎么办？

熔断器检测到连续失败，3 次后暂停 bot 并显示根因。5 分钟后自动尝试恢复。更新 key 后 bot 自动恢复，不需要重启。

这个项目会持续维护吗？

开源，MIT 协议。代码 ~7200 行 TypeScript，依赖极简（grammY + discord.js）。一个人就能维护，任何人都能 fork 和修改。

不依赖云服务、不需要订阅。即使仓库停更，你现有的部署继续运行。唯一外部依赖是 Telegram API 和 Anthropic API — 都是稳定的公共 API。
